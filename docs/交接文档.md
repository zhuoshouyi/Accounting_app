#### Step 3：数据清洗 + 明细合并 ✅

- **清洗规则**（软删除，设 `is_excluded_from_summary=1`）：
  - 状态为「还款成功」「交易关闭」「退款成功」「不计收入」→ 排除
  - 金额 ≤ 3 元（仅 expense 方向）→ 排除
- **部分退款处理**：
  - 从 raw_data JSON 的 remark 字段提取退款金额（regex 双策略：先匹配"退款"后数字，回退提取首个正数）
  - 退款 ≤ 3 元 → 状态改为「支付成功」，金额不变
  - 退款 > 3 元 → 状态改为「部分退款」，金额 = 原金额 - 退款金额
- 交互流程：按钮 → 预览（带勾选，可取消）→ 确认 → 执行（先修改后排除）
- 交易明细页面：月份筛选 + 汇总统计 + 已排除行灰色+删除线
- 4 个 Tauri 命令（累计 13 个）

#### Step 4：分类规则引擎 + 人工复核 ✅

- **三层匹配引擎**（`services/classification_engine.rs`）：
  - 第 1 层：AI 学习规则（`ai_learning_rules` 表，按 confidence DESC）→ tag_source='ai_learned'
  - 第 2 层：内置/用户规则（`category_rules` 表，按 priority ASC + match_field 排序）→ tag_source='rule'
  - 第 3 层：留空 → 进入人工复核队列
- 三种匹配方式：exact（大小写敏感）、like（大小写不敏感包含）、in（列表匹配）
- 匹配字段优先级：counterparty > product > transaction_type
- 人工复核页面：自动分类按钮 + 未分类列表 + 单条/批量指定标签
- 交易明细页面增加标签列，支持内联编辑（乐观更新+失败回滚）
- 5 个 Tauri 命令（累计 18 个）

### 3.2 分层架构约定

```
前端 (React/TS)                    后端 (Rust)
───────────────                    ────────────
pages/*.tsx     → invoke →    commands/*.rs      ← Tauri 命令层（获取 DB 连接，调用 service）
api/*.ts        ← Result ←    services/*.rs      ← 业务逻辑层
                              dao/*.rs           ← 数据访问层（纯 SQL）
                              models/*.rs        ← 数据模型
                              db/connection.rs   ← SQLite 连接管理
```

**数据流**：前端 `api/*.ts` → Tauri `invoke()` → 后端 `commands/*.rs` → `services/*.rs` → `dao/*.rs` → SQLite

### 3.3 Rust 代码规范

1. **错误处理**：所有数据库操作返回 `Result<T, String>`，前端通过 `.catch()` 捕获
2. **SQL 参数绑定**：必须用 `params![]` 或 `rusqlite::params`，禁止字符串拼接 SQL
3. **数据库连接**：通过 `State<'_, DatabaseState>` 获取，`state.conn.lock()` 获取 `MutexGuard<Connection>`
4. **Tauri 命令**：用 `#[tauri::command]` 宏标注
5. **序列化**：所有 model struct 用 `#[derive(Debug, Clone, Serialize, Deserialize)]`，字段用 snake_case（Rust 默认）
6. **模块注册**：新增模块必须在对应的 `mod.rs` 中注册
7. **命令注册**：新增 Tauri 命令必须在 `lib.rs` 的 `invoke_handler` 中用 `generate_handler![]` 注册

**Tauri 命令写法模板**（参考 `commands/cleaning_commands.rs`）：


## 五、真实账单格式（重要！）

### 5.1 微信账单

- **格式**：XLSX
- **头部**：前 17 行是头部信息，第 18 行是表头，第 19 行起是数据
- **交易时间**：datetime 类型，calamine 解析为 ExcelDateTime，需用 `as_f64()` 获取序列号后手动转换
- **Excel 日期转换**：使用 1899-12-30 作为纪元（修正 1900 闰年 bug），序列号 × 86400 秒 → NaiveDateTime
- **calamine 0.25.0 适配**：ExcelDateTime 只有 `as_f64()` 方法，没有 `as_datetime()`

### 5.2 支付宝账单

- **格式**：CSV + GBK 编码
- **头部**：前 22 行是头部，第 23 行是表头，第 24 行起是数据
- **编码**：用 `encoding_rs::GBK.decode()` 转为 UTF-8
- **CSV 解析**：每行末尾有额外逗号（多一个空列），使用 `csv::ReaderBuilder::new().flexible(true)` 处理
- **字段含 tab 字符**：解析后需 `.trim()` 处理

## 七、关键注意事项

### 7.1 开发约定

1. **增量开发**：不要修改现有文件的核心逻辑，只做增量添加。修改 `transaction_dao.rs` 等已有文件时，在文件末尾追加新方法。
2. **命令注册**：新增 Tauri 命令后，必须在 `lib.rs` 的 `invoke_handler` 中注册，否则前端调用会报错。
3. **模块注册**：新增 `.rs` 文件后，必须在对应的 `mod.rs` 中 `pub mod xxx;` 注册。
4. **前端路由**：已配置 9 条路由，新增页面需在 `App.tsx` 添加路由 + `Sidebar.tsx` 添加菜单项。
5. **serde 命名**：Rust struct 字段用 snake_case，serde 默认输出 snake_case，TS 接口字段也用 snake_case，三者保持一致。

### 7.2 已知限制

1. **重复清洗风险**：对已修改的「部分退款」交易重复执行清洗，金额可能被再次扣减。建议每批导入数据只清洗一次。后续可通过添加"已处理"标记字段优化。
2. **AI 学习规则表当前为空**：Step 4 的引擎支持查 `ai_learning_rules` 表，但该表目前为空（AI 功能在 Step 7 实现）。
3. **AG Grid / Recharts / Zustand 已在 package.json 中安装但尚未使用**：Step 6 开始使用 AG Grid，Step 8 使用 Recharts，Step 7 使用 Zustand。
4. **reqwest + aes-gcm 已在 Cargo.toml 中但尚未使用**：Step 7 开始使用。

### 7.3 用户偏好

1. **用户偏好纯文本输出**，不要 md/html 格式的报告（但代码文件和交接文档除外）
2. **用户注重数据精确性**，会仔细检查数字和规则
3. **用户用"荭包"而非"红包"**
